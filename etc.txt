Great question — this is exactly the design decision that makes or breaks Stage 6, so I'm glad you raised it before we build. Let me check what the existing specs already commit to, so my answer aligns (or consciously refines them).

I dug into the specs, and your instinct has found a real gap. Here's the situation and my recommendation.

## What the docs currently say (and where they're silent)
- **ADR-08** says *"each provision **request** owns a workspace"* — per-request language, and every example is implicitly a single VM.
- But the **lifecycle is per-VM**: `inventory` stores `workspace_path` + `terraform_state_path` **on each row**, and resize/delete/retry all say *"in the existing workspace"* (per VM).
- Nothing in the spec actually pins down the `count > 1` case. So the conflict you spotted is unresolved: per-request workspace vs per-VM lifecycle.

## Answering your exact questions
If we naïvely did **1 `main.tf` with `count = 3`** for the request, you'd hit exactly the pain you're worried about:
- All 3 VMs share **one state file**. Resizing APP02's RAM means re-planning all 3, and a bad apply can hit the siblings (blast radius).
- `count` indexes by position (`vm[0..2]`). Delete APP02 and Terraform **re-indexes** → it can destroy/recreate the wrong VM. This is the classic `count` footgun.
- A single failed VM blocks/【complicates the whole batch's retry.

## My recommendation: one workspace + state **per VM**, not per request

A `count = 3` request **fans out into 3 single-VM workspaces** at provision time:

```
storage/app/provisioning/{user}/date_pr19062026_154501/
├── APP01/   main.tf (1 VM) · tfvars · terraform.tfstate · deployment.json
├── APP02/   main.tf (1 VM) · tfvars · terraform.tfstate
└── APP03/   main.tf (1 VM) · tfvars · terraform.tfstate
```

- The **request** is the governance unit (one approval, one audit row). The **VM** is the Terraform unit (one workspace, one state, one inventory row).
- Dispatch **N `ProvisionVmJob`s** (one per VM). Partial failure is clean: 2 Active, 1 Failed → retry just that one.

**So your resize question:** increasing APP02 from 2→4 GB reuses **APP02's own workspace** — re-render *only* `APP02/terraform.tfvars` (`memory = 4096`) and `terraform apply` in place. `main.tf` never changes (it's copied from the stub and is fully variable-driven); only the tfvars differ. APP01/APP03 are never touched, no new ad-hoc `main.tf` is created, no shared state lock. That's exactly the "reconfigure CPU/RAM in existing workspace" the lifecycle doc already promises — it just has to be **that VM's** workspace.

## Why per-VM beats the alternatives
- **vs `count`**: no re-index hazard, no shared blast radius. ✅
- **vs one workspace with `for_each` map** (keyed by vm_name, per-key specs): `for_each` fixes the delete-reindex problem, and keeps a batch in one folder — but it still shares **one state file** across the 3, so a resize re-plans all of them, retry-one and per-VM destroy are messier, and one corrupt apply risks siblings. For a self-service portal where **every lifecycle action is per-VM** (renew, make-permanent, resize, add-disk, delete, retry), per-VM isolation is the cleaner match — and it's what `inventory.workspace_path`-per-row already implies.

**Honest tradeoff:** N workspaces and N applies per batch = more directories and more apply time than a single `count`/`for_each` run. Mitigation: the per-VM jobs can run in parallel on the queue. For governance + clean lifecycle, the isolation is well worth it — this is the standard pattern for VM self-service portals.

## What I'd do before building Stage 6
This needs a small **ADR-08 amendment**: change *"per-request workspace"* → *"per-VM workspace (a request with `instance_count = N` fans out to N workspaces/jobs/inventory rows; the request remains the single approval/audit unit)"*, and note resize/add-disk/delete/retry operate on that VM's workspace. Then Stage 6 builds to that.
